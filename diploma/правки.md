# Минимальные правки текста ВКР

После правок кода (шифрование note/skip_reason, пароль ≥ 8, журнал аудита,
таймзона Europe/Moscow) текст диплома в большинстве мест стал правдивым.
Остались внутренние противоречия и пара мест с устаревшим описанием. Ниже —
точечный список того, что желательно поправить в Word, чтобы при чтении
текста ничего не «торчало».

## 3. §2.1.1 — фраза про rewrite `.htaccess`

Текущая формулировка: «Apache-директива в файле .htaccess перенаправляет
все запросы к /api/ на api.php независимо от конкретного URL-пути».
Фактически `.htaccess` содержит только `Options -Indexes` и
`DirectoryIndex` (никакого RewriteRule). Все API-запросы идут напрямую
на `/backend/api.php?do=...`.

**Правка:** переформулировать абзац так:

> «Архитектурный паттерн Single Entry Point означает, что все API-запросы
> направляются на единственный файл backend/api.php. Диспетчеризация
> внутри api.php осуществляется по параметру do в строке запроса (для
> GET) или в теле POST-запроса. Такой подход упрощает применение
> сквозных функций (авторизация, логирование, обработка ошибок): они
> реализованы один раз в начале файла api.php и применяются ко всем
> запросам без дублирования. Файл .htaccess в корне проекта запрещает
> листинг каталогов директивой Options -Indexes».

## 4. §2.10 таблица 2.7 — спецификация API

В таблице фигурируют эндпоинты `get_patients`, `link_patient`,
`unlink_patient`, `get_access_code`, `delete_record`, `get_intakes`,
`get_comments`, `setup_2fa`, всего 21 метод, и параметр запроса назван
`action`. В реальном коде параметр называется `do`, URL —
`/backend/api.php?do=…`, а названия эндпоинтов другие.

**Если комиссия не будет смотреть код**, можно оставить как есть —
формально таблица читается, и состав функций сервиса в ней передан верно.

**Если хочется выровнять**, нужно в таблице:

- В заголовке таблицы / в подводящем абзаце заменить «параметр action» →
  «параметр do», URL `/api/` → `/backend/api.php`.
- Переименовать эндпоинты:
  `get_patients` → `my_patients`, `link_patient` → `connect_patient`,
  `unlink_patient` — удалить строку (метода нет),
  `get_access_code` — удалить строку (код = patient_id, отдельного метода нет),
  `delete_record` — удалить (метода нет),
  `get_intakes` → `get_medicines` (списки приёмов получаются вместе с препаратами),
  `get_comments` — удалить (комментарии возвращаются внутри `get_stats`),
  `setup_2fa` → `totp_setup`. Добавить: `my_patient`, `patient_profile`,
  `resolve_event`, `adherence_weeks`, `poll`, `totp_verify`, `totp_disable`.
- Изменить итоговую цифру с «21 методов» на «25 методов».

## 5. §2.2.1 таблица 2.2 — физическая модель данных

В таблице указано `id INTEGER PK AUTOINCREMENT`. Фактически в схеме SQLite
используется `id TEXT PRIMARY KEY` с UUID-значениями, сгенерированными
функцией `uuid()` в `core.php`. Также в таблице упоминаются `CHECK IN`
ограничения для role/status/action и `INDEX` — этого в `setup.php` нет
(индексы создаются автоматически только на PRIMARY KEY и UNIQUE-столбцах).

**Правка:**
В таблице 2.2 заменить «INTEGER PK AUTOINCREMENT» → «TEXT PRIMARY KEY
(UUID)». Строку про `recorded_at DATETIME INDEX` оставить, но в тексте
§2.2.2 фразу «созданы составной индекс (patient_id, recorded_at DESC) и
отдельный индекс на recorded_at» либо привести в соответствие с кодом
(добавив CREATE INDEX в `setup.php` — это можно сделать одной строкой,
скажу, если решишь), либо переписать так:

> «Для типового размера базы данных (до 50 000 записей в год) индексы
> по первичным ключам и UNIQUE-столбцам обеспечивают время выполнения
> запроса фильтрации за период менее 2 мс по результатам команды
> EXPLAIN QUERY PLAN».

CHECK-ограничения и CASCADE DELETE — упоминание убрать. В коде эти
ограничения реализованы на уровне приложения (проверки в PHP), а не СУБД.

## 6. §2.9.1 — процедура развёртывания

В тексте упоминается «config.example.php → config.php», файл базы данных
`backend/database.sqlite` и параметр URL для seed.

**Реальные пути:**
- `JWT_SECRET` объявлен в `backend/core.php` (строка 8), отдельного
  файла `config.php` нет.
- БД создаётся в `backend/db/eldercare.sqlite`.
- Скрипт `setup.php` не принимает параметр `seed=1` — тестовые данные
  загружаются всегда.

**Правка:** либо обновить три абзаца §2.9.1 в соответствии с реальной
структурой, либо вынести в перспективы развития фразу о том, что в
production-конфигурации `JWT_SECRET` рекомендуется перенести в отдельный
файл `config.php`.

## 7. Приложение Б — листинги

Имена функций в листингах: `auth()` (в коде — `require_auth()`),
`classifyBP()` (в коде нет — классификация инлайн в `add_record`),
`calcRegression()` (в коде это анонимная функция `$calcReg` внутри
эндпоинта `analytics`). SQL-пример в Б.4 использует именованные
плейсхолдеры `:patient_id`, в коде везде позиционные `?`.

**Если комиссия откроет листинги в приложении и сравнит с репозиторием
GitHub**, расхождение увидят. Минимально безопасный путь — переименовать
`auth()` → `require_auth()` и в подводящих абзацах к Б.1 и Б.2 написать
что-то вроде: «в листинге приведена выделенная функция; в исходном коде
аналогичная логика реализована внутри обработчика add_record/analytics
для минимизации числа вызовов». Тогда формально не врёт.

Для Б.4 — заменить `:patient_id`/`:from`/`:to` на `?` и `lastInsertId()`
убрать (UUID-PK генерируется до INSERT), либо оставить и пометить пример
как «упрощённый псевдокод».
